Business Rules

Customer Request & Payment (BR-001 to BR-014)
  • BR-001 Request cannot be transferred to paid pipeline >1 time. Archived after transfer.
  • BR-002 Cannot update/delete transferred request.
  • BR-003 otherCategoryDescription required if category='other'.
  • BR-004 Phone format validation required.
  • BR-005 Proof files mandatory on transfer.
  • BR-006 Status: pending → verified|rejected|request_clarification; verified → approved (CEO).
  • BR-007 Only Finance can verify/reject/clarify.
  • BR-008 Only CEO executes final approval.
  • BR-009 Clarification requires mandatory message.
  • BR-010 Rejection requires mandatory message.
  • BR-011 All decisions logged in finance_history.
  • BR-012 Clarification response requires description + ≥1 image.
  • BR-013 Marketing responds only when status=request_clarification.
  • BR-014 History & responses are immutable audit snapshots.
Designer Workflow (BR-015 to BR-027)
  • BR-015 Strictly sequential phases. No skipping.
  • BR-016 Rejected phase locks task.
  • BR-017 Feedback requires revision to restart review.
  • BR-018 Cannot proceed if current phase = feedback.
  • BR-019 History preserved as immutable snapshots.
  • BR-020 Latest live data stored separately.
  • BR-021 Revisions append new snapshots.
  • BR-022 Phase rejection → task = rejected.
  • BR-023 All 4 approved → task = completed, approvalStatus = approved.
  • BR-024 Final Stage screenshot mandatory.
  • BR-025 Only submit to assigned tasks.
  • BR-026 Performance reviews only for completed tasks.
  • BR-027 Ratings 0–5, 1 decimal max.
Applications, QS, Tasks & Users (BR-028 to BR-067)
  • BR-028-033 Applications: 1 per posting, assigned → task created, CEO/GM review only.
  • BR-034-043 QS: pending_review → in_review → record_submitted. Independent read tracking.
  • BR-044-051 Tasks: pending → in_progress → end states. Approve optional feedback, reject mandatory.
  • BR-052-058 Users: CEO only CRUD roles. Unique username, min 8 char pass, valid email, 8 allowed roles.
  • BR-059-067 Projects & Audit: Stages/Status constraints. Every state change appended to audit logs. CEO only view.

Notifications & Non-Functional Requirements

Notification Principles

  • Role-targeted via target_roles array.
  • Read status tracked per-user in notification_reads. User A reading ≠ User B.
  • Counts calculated server-side via aggregate queries.
  • Bulk mark-read & role-specific type filtering supported.

Security

  • JWT Bearer, RBAC middleware
  • bcrypt/argon2id, hashed refresh
  • Input sanitization, rate limiting
  • File MIME validation, presigned URLs

Performance

  • Pagination on all lists
  • <500ms reads, <1s writes
  • Indexes on FKs & filters
  • Streaming uploads/downloads

Reliability

  • Transactional multi-table ops
  • Atomic assignment → task
  • Soft deletes & audit trails
  • Stateless API design
