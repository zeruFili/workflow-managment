📋 1. Project Overview

🎯 Purpose

The Workflow Management System orchestrates the full operational lifecycle: customer intake → payment verification → project creation → task assignment → multi-phase design production → quantity/cost review → site execution → performance evaluation.

✅ Goals
  1. Single source of truth for operational data across 8 roles.
  2. Role-based access control (RBAC) at the API layer.
  3. Full audit history for every state-changing action.
  4. Multi-role notification with independent read-status per user.
  5. Gated, linear workflow progression (4-phase designer pipeline).
  6. REST API consumed by a Telegram Mini App frontend.
🔄 Main Workflow
📥 Customer Intake → 💳 Payment Upload → ✅ Payment Verification → 👑 CEO Approval → 🏗 Project Creation → 📝 Job Posting → 🎨 Designer Application → 📐 Assignment → 🖌 Multi-Phase Design → 🔍 Phase Review → 📊 QS Review → 🎯 Leadership Decision → 🏗 Site Execution → ⭐ Performance Rating

👥 2. User Roles

Eight distinct roles power the system, each with scoped responsibilities and access.

ceo
👑 Chief Executive Officer

Full oversight, user management, finance decisions, escalation authority.

general_manager
📋 General Manager

Operational leadership, task assignment, approval review. Mirrors CEO minus audit/user CRUD.

marketing_lead
📢 Marketing Lead

Customer intake, paid-customer handling, clarification responses to finance.

finance_officer
💰 Finance Officer

Payment verification, rejection, clarification requests. Gatekeeper of the payment pipeline.

data_collector
📸 Data Collector

Field data capture, evidence submission for tasks.

quantity_surveyor
📊 Quantity Surveyor

Cost/quality evaluation of design submissions. Recommends approval or revision.

designer
🎨 Designer

Job applications, multi-phase design submission (Case Study → Design → Rendering → Final).

site_engineer
🏗 Site Engineer

Field task execution, status updates, evidence capture at construction sites.

🔔 5. Notification Rules

Role-targeted notifications with independent per-user read tracking.

📌 Key Principles
  • NOT-001: Notifications are role-targeted via target_roles array.
  • NOT-002: Read status tracked per-user — User A reading does NOT affect User B.
  • NOT-003: notification_reads records (user_id, notification_id, read_at).
  • NOT-004: Mark read individually or in bulk.
  • NOT-005: Notification counts calculated as unread for authenticated user's role.

⚡ 10. Non-Functional Requirements

🔒 Security (NFR-SEC)
  • JWT Bearer on all endpoints (except login/refresh). Access token: 15 min. Refresh: 7 days hashed.
  • RBAC middleware validates role before business logic.
  • Passwords: bcrypt (cost≥12) or argon2id. Never log plaintext.
  • Input sanitized — parameterized queries only. File uploads validated MIME + max 10MB.
  • Rate limiting: 5 login attempts/IP/minute. Security headers enforced.
📊 Performance (NFR-PERF)
  • Pagination on all list endpoints (page, perPage; default 20, max 100).
  • 95th percentile: <500ms reads, <1000ms writes.
  • Indexes on all FKs and filter columns. Streaming for file upload/download.
📝 Logging & Audit (NFR-LOG)
  • Every state change logged: entity, ID, action, actor, timestamp, old/new values (JSONB).
  • Audit logs append-only. Only CEO views. No passwords/tokens in logs.

💡 11. Assumptions & Recommendations

📌 Key Assumptions
  1. JWT Bearer with role claims. Frontend currently uses mock auth.
  2. Single-tenant deployment. No tenant isolation needed.
  3. In-app notifications only; no email infrastructure.
  4. REST GET polling for notifications; no WebSocket/SSE.
  5. Assigned application creates designer task (one-to-one).
  6. DC tasks can share unified tasks table with discriminator column.
🔮 Top Recommendations
  1. REQ-REC-001: Implement forgot/reset password flow with email token.
  2. REQ-REC-003: JWT blacklist for immediate invalidation on logout/role change.
  3. REQ-REC-007: Automated deadline reminder notifications (<24h, <1h).
  4. REQ-REC-010: Include /api/v1/ prefix for versioning.
  5. REQ-REC-013: Health check endpoint GET /api/health.
  6. REQ-REC-015: Soft delete consistency across all entities.
  7. REQ-REC-018: Allow QS resubmission cycle on feedback.
  8. REQ-REC-020: Full-text search via PostgreSQL tsvector.
